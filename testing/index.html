    üß™ Spring Boot Integration Tests & Database Lifecycle

    This document explains how Spring Boot manages the database lifecycle during integration tests ‚Äî specifically when the database is created, when it's dropped, and how isolation is achieved.


    > üõ¢Ô∏è Database Creation and Dropping

    >> When is the database created?

    When using an in-memory database like H2 and spring.jpa.hibernate.ddl-auto=create-drop:

    • ‚úÖ The database schema is created once, when the application context is first started.
    • This usually happens once per test class, if you're using @SpringBootTest.
    • The same context (and DB schema) is reused across test methods in the same class.
    • ‚ùå The database is not recreated before each test method.

    >> When is the database dropped?

    • With ddl-auto=create-drop, the schema is dropped automatically when the Spring context shuts down, i.e.:
    • After the test class finishes (if context was specific to that class)
    • Or at the end of the full test suite (if context was reused)



    > ‚ôªÔ∏è Transactional Rollback and Isolation

    >> What does @Transactional do?

    When a test class or test method is annotated with @Transactional:

    • Each test method runs in its own transaction
    • After the test method completes, the transaction is rolled back
    • This leaves the database in a clean state between tests ‚Äî without recreating the schema

    >> Example

    @SpringBootTest
@Transactional
class MyIntegrationTest {

    @Autowired
    UserRepository userRepository;

    @Test
    void testCreateUser() {
        userRepository.save(new User("Alice"));
        // Rolled back after the test method finishes
    }
}



    > üîé Controlling DB and Context Scope

    Situation Context Reused? DB Recreated? Notes
    Multiple tests in same class ‚úÖ Yes ‚ùå No Same context, same schema
    Multiple test classes (default) ‚úÖ Yes ‚ùå No Spring optimizes by reusing context
    With @DirtiesContext ‚ùå No ‚úÖ Yes Forces full context + DB reset
    With @Transactional ‚úÖ Yes ‚ùå No Rollback used instead of full reset



    > üëç Best Practices for Integration Tests

    ‚úÖ Use @Transactional to ensure clean state without heavy DB recreation
    ‚úÖ Use ddl-auto=create-drop for in-memory DBs (e.g. H2) to keep schema fresh
    ‚úÖ Use @DirtiesContext only when you need full context isolation
    ‚úÖ Use fixture loaders (e.g. JSON or SQL) to control test data explicitly
    ‚ö†Ô∏è Avoid ddl-auto=update in tests ‚Äî unpredictable and risky
    ‚ö†Ô∏è Avoid mixing stateful and stateless test strategies in the same class

    üìå Summary

    • Spring Boot creates the DB once per context, usually once per test class.
    • Schema is created at context startup, dropped at context shutdown.
    • Use @Transactional for method-level isolation.
    • Use @DirtiesContext if you need class-level isolation with fresh schema.
